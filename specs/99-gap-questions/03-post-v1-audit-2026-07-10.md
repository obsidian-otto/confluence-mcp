## Issue 1 — `conf_post` schema/body mismatch (BLOCKER for writes)

### Symptom

Calling the MCP tool:

    mcp__confluence__conf_post(
        path="/wiki/api/v2/pages",
        body={"spaceId":"780763211","status":"current",
              "title":"mcp-test-page","body":{"representation":"storage","value":"<p>Hello</p>"}})

yields from upstream:

    POST /wiki/api/v2/pages: 400 Bad Request — {"errors":[{"status":400,"code":"INVALID_MESSAGE","title":"Invalid message","detail":null}]}

### Diagnosis

Two layers mismatch:

1. The MCP tool descriptor for `conf_post` (auto-generated by
   `metoro-io/mcp-golang` from the `PostArgs` struct) declares
   `body` as an ARRAY of object: `body: { type: array, items: { type: object, properties: {...} } }`.

2. The Go-side `PostArgs.Body` field in `internal/tools/args.go` is
   `map[string]any` — a single object. The Phase 5 agent flagged
   this asymmetry in the kickoff but only resolved PATCH's
   `[]map[string]any` shape per the spec.

3. The handler chain (`HandlePost` → `executeRequest` →
   `atlassian.Client.Do`) calls `json.Marshal(a.Body)`. When the
   caller passes an empty array (mcp-golang's default if no `body`
   param is sent), the handler marshals `[]` which the upstream
   Confluence v2 API rejects with the generic `INVALID_MESSAGE`.

When the caller supplies an array-shaped body via the MCP tool, the
JSON unmarshal into `PostArgs.Body` (an object) fails, leaving
`Body` as nil and again sending an empty body upstream.

### Suggested fix (NOT applied — record only)

Two viable approaches:

**A. Change the MCP descriptor, not the Go type** — at register
time in `internal/tools/register.go`, wrap each per-tool handler so
the schema declares `body` as a single object (not array) for
`conf_post` / `conf_put` / `conf_patch`.

**B. Change the Go type** — change `PostArgs.Body` and `PutArgs.Body`
from `map[string]any` to `[]map[string]any` and at handler time
either `bytes.NewBuffer(json.Marshal(a.Body[0]))` (for exactly one
element) or join them.

Approach **A** is preferred because:
- The upstream (`@aashari/mcp-server-atlassian-confluence`) takes
  `body` as a single object for POST/PUT (only PATCH uses an
  array).
- The Phase 5 types already match the upstream (`map[string]any`
  for POST/PUT).
- mcp-golang's introspection is the layer that's wrong; correcting
  the handler registration is local to `register.go` and does not
  drift the type or any tests.

If A is taken, the matching change is in
`internal/tools/register.go::RegisterAll` — wrap each handler's
arg type in a per-method struct so the schema is one we control
rather than one auto-derived from `PostArgs.Body` being
`map[string]any` but introspected as a slice somehow.

### Why this isn't blocking Phase 12

- The end-to-end smoke test (Phase 10) successfully completed
  `tools/list` and one `tools/call conf_get`. POST was not on the
  Phase 10 critical path; it was deferred to "real-world usage".
- The Hermes-side MCP framework correctly surfaces the upstream
  400 — the error envelope is propagated verbatim. That's a
  PASS for the **anti-pattern spec** in `specs/09-anti-patterns/03-error-shapes.md`
  (literal shape: `<METHOD> <path>: <status> <text> - <body>`).
  Verified: error propagation works; the issue is upstream
  rejection of an empty body.

## Issue 2 — Direct-stdin invocation produces empty stdout (EXPECTED, but worth noting)

When invoking the binary directly via `cat payload.json | bin/mcp-confluence`,
the binary produces no stdout. Cause is by design:
`wireStdinEOF` in `cmd/mcp-confluence/main.go` uses an `io.Pipe`
gated by an `io.Copy` goroutine. When `cat` closes its stdout,
the binary's stdin hits EOF, the goroutine cancels the context,
and the transport's readLoop exits before fully servicing the
queued `initialize` request.

This is correct MCP stdio behavior — parents (Hermes) keep stdin
open. Documented here so future maintainers don't waste time
chasing it.

## Issue 3 — MCP auto-retry rate limit (transient, observed)

After 4–5 consecutive failures on `conf_post` (due to issue #1),
the Hermes MCP framework marked `confluence` as unreachable with
"Auto-retry available in ~50s. Do NOT retry this tool yet".
The framework auto-recovers; no action needed.

This is informational only — a future `hermes mcp` improvement
might surface unrate-limited error context for known MCP error
shapes so the human can fix the upstream issue without waiting.

## Suggestions for future v1.x

1. **Add a `/docs` or `/mcp-help` tool** so MCP servers can self-
   describe their body shape constraints. Would have saved the
   `conf_post` loop in this audit (we hit 4 identical 400s
   before moving on).

2. **Improve Hermes `mcp test <name> <tool> ...` to pass argument
   JSON** so end-to-end smoke tests via the Hermes CLI don't
   require round-tripping through the full MCP framing layer.
   Today's `hermes mcp test confluence` only lists tools; an
   optional `--call <tool> '<json-args>'` would let operators
   verify a server in 5 seconds instead of waiting for the
   auto-retry cooldown.

3. **Surface Confluence schema hints inline** — the upstream
   error `INVALID_MESSAGE:detail=null` is unhelpful. A
   server-side handler could detect 400 from POST/PATCH paths
   and prepend a sentence describing the expected body shape.
