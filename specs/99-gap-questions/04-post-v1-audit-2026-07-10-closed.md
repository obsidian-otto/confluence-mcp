## Issue 1 — `conf_post` schema/body mismatch (CLOSED via explicit `jsonschema:` tags)

### Original symptom

Calling the MCP tool:

    mcp__confluence__conf_post(
        path="/wiki/api/v2/pages",
        body={"spaceId":"<numeric-space-id>","status":"current",
              "title":"mcp-test-page","body":{"representation":"storage","value":"<p>Hello</p>"}})

yielded from upstream:

    POST /wiki/api/v2/pages: 400 Bad Request — {"errors":[{"status":400,"code":"INVALID_MESSAGE","title":"Invalid message","detail":null}]}

### Original diagnosis (now corrected)

**The original audit doc stated:** "the MCP tool descriptor for
`conf_post` (auto-generated by metoro-io/mcp-golang from the
`PostArgs` struct) declares `body` as an ARRAY of object: `body:
{ type: array, items: { type: object, properties: {...} } }`."

**That diagnosis was wrong.** Re-introspecting the schema after
adding explicit `jsonschema:` tags on every field:

```json
"body": {
  "type": "object",
  "description": "JSON object to send as the request body. ..."
}
```

`PostArgs.Body` (and `PutArgs.Body`) are `map[string]any`. The
`invopop/jsonschema@v0.12.0` `reflectMap` method produces
`type: object` for Go `map` types regardless of the value side.
**There was never an `items: object` issue.** The schema was
correct from day one.

What WAS wrong was that the user-visible description for `body`
was implicit (no `jsonschema:` tag), so an MCP client reading the
schema had no way to know what shape to send. Empty implicit
`type: object` schemas with no description look like "send any
object", which is technically true but unhelpful.

### Fix shipped (commit pending)

Every args-struct field now carries a non-empty
`jsonschema:"description=...,required"` tag. The description
includes an explicit example — e.g.:

    Body         map[string]any    `json:"body,omitempty" jsonschema:"description=JSON object to send as the request body. For creating a Confluence page: {\\\"spaceId\\\":\\\"<numeric>\\\",\\\"status\\\":\\\"current\\\",\\\"title\\\":\\\"My title\\\",\\\"body\\\":{\\\"representation\\\":\\\"storage\\\",\\\"value\\\":\\\"<p>HTML content</p>\\\"}}. Object-only; use a single object, never an array, for POST endpoints."`

`PatchArgs.Body` (which IS `[]map[string]any`) keeps the array
shape — its schema correctly shows `type: array, items:
{type: object}`.

### Tests that lock the fix in

| Test | Asserts |
| ---- | ------- |
| `TestArgsJsonschemaTagsPresent` | every field of every args struct has a non-empty `jsonschema:` tag |
| `TestArgsSchemasAreAccurate` | for each args struct: schema is type:object with properties; at least one description is populated; required fields (path / page-id / cql) are marked; body type is `object` for POST/PUT and `array` for PATCH |

Both tests use `github.com/invopop/jsonschema` (the same
library mcp-golang v0.16.1 uses internally) so they reflect
exactly what the wire sees.

### Empty body upstream rejection

The upstream Confluence v2 API rejects an empty body for
`POST /wiki/api/v2/pages` with `INVALID_MESSAGE`. The fix does
NOT make empty bodies valid; it makes the schema clear about
what shape to send. The next audit should rerun the original
Failing call (above) end-to-end with the new binary and confirm
the upstream accepts the page-create body. **Action queued for
operator verification.**

## Issue 2 — Direct-stdin invocation produces empty stdout (EXPECTED, in-progress)

When invoking the binary directly via `cat payload.json | bin/mcp-confluence`,
the binary produces no stdout. Cause is by design:
`wireStdinEOF` in `cmd/mcp-confluence/main.go` uses an `io.Pipe`
gated by an `io.Copy` goroutine. When `cat` closes its stdout,
the binary's stdin hits EOF, the goroutine cancels the context,
and the transport's readLoop exits before fully servicing the
queued `initialize` request.

This is correct MCP stdio behavior — parents (Hermes) keep stdin
open. Documented here so future maintainers don't waste time
chasing it.

## Issue 3 — MCP auto-retry rate limit (transient, in-progress)

After 4–5 consecutive failures on `conf_post` (due to issue #1),
the Hermes MCP framework marked `confluence` as unreachable with
"Auto-retry available in ~50s. Do NOT retry this tool yet".
The framework auto-recovers; no action needed.

## Quality-of-life improvements shipped

The audit recommended adding helpers "to directly get all
workspaces etc." The five new tools are now registered with the
running MCP server:

| Tool | Path | Args summary |
| ---- | ---- | ------------ |
| `conf_list_spaces` | GET /wiki/api/v2/spaces | `limit`, `cursor`, `type`, `status`, `outputFormat` |
| `conf_list_pages` | GET /wiki/api/v2/pages | `space-id`, `space-key`, `title`, `status`, `limit`, `cursor`, `sort`, `body-format`, `outputFormat` |
| `conf_get_page_body` | GET /wiki/api/v2/pages/{id}/body | `page-id` (required), `body-format`, `outputFormat` |
| `conf_search` | GET /wiki/rest/api/search | `cql` (required), `limit`, `start`, `excludedContent`, `outputFormat` |
| `conf_help` | local (no network) | `topic`, `outputFormat` |

Each new tool delegates to `executeRequest` so it inherits the
9-step pipeline (TOON encoding, JMESPath, 40k truncation, error
shape). The `conf_help` tool is local-only — its response is
synthesised from the registered `CONF_*_DESCRIPTION` constants
so it works even when Confluence is unreachable.
